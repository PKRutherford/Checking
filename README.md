### 5. What is an Incident and how do you handle it?
**Answer:**
An incident is an unexpected interruption or reduction in quality of an IT service. 

**Incident Handling Process:**
1. **Identification:** Detect and log the incident
2. **Classification:** Categorize by priority and severity
3. **Investigation:** Analyze logs and identify root cause
4. **Resolution:** Apply fix or workaround
5. **Closure:** Document and close the incident
6. **Follow-up:** Ensure the issue doesn't recur

---

### 6. Explain the difference between Severity and Priority
**Answer:**

**Severity:** Impact of the issue on the system
- **Critical:** System down, affects all users
- **High:** Major functionality affected
- **Medium:** Partial functionality affected
- **Low:** Minor issue, cosmetic

**Priority:** Urgency of fixing the issue
- **P1:** Immediate fix required
- **P2:** Fix within a few hours
- **P3:** Fix within a day
- **P4:** Fix when possible

Example: A spelling mistake in production might have low severity but high priority if it's on a customer-facing page.

---

### 7. What is SLA (Service Level Agreement)?
**Answer:**
SLA is a commitment between a service provider and a client that defines:
- Expected service quality levels
- Response times for different priority incidents
- Availability guarantees (e.g., 99.9% uptime)
- Metrics for measuring performance
- Consequences if SLA is not met

Example: P1 incidents must be responded to within 15 minutes and resolved within 2 hours.

### 8. How do you troubleshoot an application issue?

1. **Gather Information:** Understand the error message, when it occurred, and who is affected
2. **Check Logs:** Review application logs, system logs, and error logs
3. **Reproduce the Issue:** Try to replicate the problem
4. **Identify Root Cause:** Analyze patterns and dependencies
5. **Apply Fix:** Implement solution or workaround
6. **Test:** Verify the fix works
7. **Document:** Record the issue and resolution for future reference

### 15. Explain Client-Server Architecture
**Answer:**
Client-Server architecture is a computing model where:

**Client:**
- Requests services/resources
- User interface (browser, app)
- Sends requests to server

**Server:**
- Provides services/resources
- Processes requests
- Sends responses back to client

Example: When you access a website, your browser (client) requests a web page from the web server, which processes the request and sends back the HTML content.

### 23. What are Different Types of Backups?
**Answer:**
- **Full Backup:** Complete copy of all data
- **Incremental Backup:** Only changes since last backup
- **Differential Backup:** Changes since last full backup
- **Mirror Backup:** Exact replica of source data
- **Cloud Backup:** Data stored in cloud storage

### 24. What is Root Cause Analysis (RCA)?
**Answer:**
RCA is a systematic process of identifying the fundamental cause of a problem to prevent recurrence. 

**Steps:**
1. Define the problem
2. Collect data and evidence
3. Identify possible causes
4. Determine root cause
5. Implement corrective actions
6. Monitor effectiveness

Tools: 5 Whys, Fishbone Diagram, Fault Tree Analysis

### 40. Explain OSI Model
**Answer:**
The OSI (Open Systems Interconnection) model has 7 layers:

1. **Physical:** Hardware transmission (cables, switches)
2. **Data Link:** Frame transmission (MAC addresses)
3. **Network:** Routing (IP addresses)
4. **Transport:** End-to-end communication (TCP/UDP)
5. **Session:** Session management
6. **Presentation:** Data formatting (encryption)
7. **Application:** User interface (HTTP, FTP)

## TECHNICAL SCENARIOS

### 46. If a critical production server goes down at 2 AM, what would you do?
**Answer:**
1. **Acknowledge:** Log the incident immediately in ticketing system
2. **Assess:** Check monitoring tools for error messages and logs
3. **Escalate:** Inform senior team members and stakeholders
4. **Investigate:** Identify if it's hardware, network, or application issue
5. **Restore:** Attempt to restart services or failover to backup
6. **Communicate:** Update status to management and affected users
7. **Document:** Record all actions taken
8. **RCA:** Perform root cause analysis after restoration

---

### 47. How do you prioritize multiple incidents?
**Answer:**
Priority based on:
1. **Business Impact:** Number of users affected
2. **Severity:** Critical systems vs. minor features
3. **SLA Requirements:** Time-sensitive commitments
4. **Dependencies:** Issues blocking other work

Example: A login failure affecting all users (P1) takes priority over a report formatting issue (P3).

---

### 48. What would you do if you don't know the solution to a problem?
**Answer:**
1. **Check Documentation:** Review knowledge base and previous tickets
2. **Search:** Google the error message and symptoms
3. **Consult:** Ask senior team members or subject matter experts
4. **Escalate:** If urgent, escalate to L2 or L3 support
5. **Learn:** Document the solution for future reference
6. **Follow Up:** Understand the root cause to handle similar issues independently
